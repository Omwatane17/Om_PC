SYSTEM OVERVIEW
System Overview & Architecture

1.1 Purpose
PradnyaChakshu  is a multi-tier web application that ingests decision datasets, computes statistical fairness metrics, generates explainability artefacts via SHAP analysis, and synthesises plain-English bias narratives using Claude 3 Sonnet. The system is designed to be stateless and horizontally scalable, with all long-running analysis tasks handled asynchronously via a distributed task queue.

1.2 High-Level Architecture
The system follows a three-tier architecture with an additional async processing layer:

Tier	Components	Responsibility
Presentation	React SPA (TypeScript)	User interface, file upload, dashboard rendering, report download
API Gateway	FastAPI (Python 3.11)	REST endpoints, request validation, auth, WebSocket channels
Task Queue	Celery + Redis	Async execution of fairness analysis, SHAP computation, LLM calls
ML Engine	scikit-learn, Fairlearn, AIF360, SHAP	Metric computation, model wrapping, feature attribution
LLM Layer	Anthropic Claude 3 Sonnet API	Natural language explanations, remediation suggestions
Data Store	PostgreSQL + Redis	Persistent audit records, task state, session caching
Object Store	Local FS → S3-compatible	Uploaded datasets, generated PDF reports

1.3 Architectural Principles
•	Separation of Concerns: ML computation is fully decoupled from API layer via Celery
•	Fail-Safe Design: Fairness engine errors degrade gracefully; partial results are returned with warnings
•	Privacy by Design: Uploaded data is never persisted beyond session TTL (24 hours default)
•	Idempotency: All audit jobs are idempotent; re-running same dataset produces identical results
•	Observability: Structured JSON logging at every layer; Prometheus metrics exposed on /metrics

ARCHITECTURE DEEP DIVE
System Architecture Deep Dive

3.1 Request Lifecycle
1.	Client uploads CSV via multipart/form-data POST to /api/v1/audits
2.	FastAPI validates file (size, MIME type, row count). Returns 400 on violation.
3.	File is written to ephemeral object store (UUID-namespaced path).
4.	A Celery task chain is enqueued: [detect_attributes | compute_metrics | run_shap | call_llm | generate_pdf]
5.	API returns 202 Accepted with { audit_id, websocket_channel }
6.	Client subscribes to WebSocket channel. Each Celery task emits progress events.
7.	On completion, final audit result is stored in PostgreSQL and client receives DONE event.
8.	Client fetches full result via GET /api/v1/audits/{audit_id}

3.2 Celery Task Chain
Tasks are chained using Celery's chord/chain primitives. Each task is independently retryable:

# tasks/audit_chain.py
from celery import chain, chord
 
def create_audit_chain(audit_id: str, file_path: str):
    return chain(
        detect_sensitive_attributes.s(audit_id, file_path),
        compute_fairness_metrics.s(),
        chord(
            [run_shap_analysis.s(), compute_counterfactuals.s()],
            aggregate_explanations.s()
        ),
        call_llm_explainer.s(),
        generate_pdf_report.s(),
    ).apply_async()

3.3 WebSocket Event Schema
All WebSocket messages follow a typed envelope:

// TypeScript types (shared between client and server via openapi-typescript)
interface WSEvent {
  audit_id:  string;
  event:     'progress' | 'partial_result' | 'done' | 'error';
  stage:     'detect' | 'metrics' | 'shap' | 'llm' | 'pdf';
  pct:       number;          // 0–100
  message:   string;
  payload?:  Partial<AuditResult>;
}

3.4 Data Isolation Model
Each audit session is fully isolated:
•	Uploaded files are stored under /tmp/PradnyaChakshu /{audit_id}/ and deleted after 24 hours via a cron job.
•	PostgreSQL rows are partitioned by organisation_id (UUID v4 per registration).
•	Redis keys are namespaced as PradnyaChakshu :{org_id}:{audit_id}:{key}.
•	No cross-tenant data leakage is possible; foreign-key constraints enforce org boundaries.

CORE ALGORITHMS & ML MODULES
Core Algorithms & ML Modules

6.1 Sensitive Attribute Detector
The detector uses a three-stage pipeline to identify protected attributes without requiring user specification:

Stage	Method	Examples Detected
1. Column Name Heuristics	Regex + keyword list match against column names (case-insensitive)	gender, sex, race, ethnicity, age, dob, zip, postcode, nationality
2. Value Distribution Analysis	High-cardinality columns excluded; low-cardinality string/int columns flagged	Columns with 2–20 unique values in small alphabets
3. Embedding Similarity	Column name embedded via sentence-transformers all-MiniLM-L6-v2; cosine similarity >0.75 to protected-attr seed set	Catches obfuscated names: 'demo_group', 'segm_code'

6.2 Fairness Metrics Engine
All metrics are implemented using Fairlearn 0.10 as the primary library, with custom implementations for metrics not covered. Each metric is computed per protected attribute and per intersectional group pair.

Metric	Formula (simplified)	Pass Threshold	Regulation
Demographic Parity Difference	P(ŷ=1|A=a) − P(ŷ=1|A=b)	|value| ≤ 0.10	EU AI Act
Equalized Odds Difference	max(TPR gap, FPR gap) across groups	|value| ≤ 0.10	EEOC
Equal Opportunity Difference	TPR(A=a) − TPR(A=b)	|value| ≤ 0.10	EEOC
Predictive Parity Difference	PPV(A=a) − PPV(A=b)	|value| ≤ 0.10	CFPB
Disparate Impact Ratio	P(ŷ=1|A=a) / P(ŷ=1|A=b)	≥ 0.80 (4/5 rule)	EEOC / CFPB
Calibration Difference	|P(Y=1|ŷ=s,A=a) − P(Y=1|ŷ=s,A=b)|	|value| ≤ 0.05	Generic
Individual Fairness Score	1 − avg Lipschitz violation over sampled pairs	≥ 0.90	EU AI Act
Counterfactual Fairness Score	P(ŷ_CF=1) vs P(ŷ=1) after attr flip	Ratio in [0.9, 1.1]	EU AI Act
Balance (Positive Class)	avg score | Y=1 equal across groups	Std dev ≤ 0.05	Generic
Balance (Negative Class)	avg score | Y=0 equal across groups	Std dev ≤ 0.05	Generic
Treatment Equality	FN/FP ratio equal across groups	Max ratio diff ≤ 0.20	Generic
Intersectional Disparity	Pairwise DPD across all attr combinations	|value| ≤ 0.15	EU AI Act

6.3 SHAP Explainability Module
SHAP (SHapley Additive exPlanations) is used to attribute prediction contributions to input features. The explainer type is chosen based on model type:

# engine/shap_module.py
import shap
import numpy as np
 
class PradnyaChakshu ShapExplainer:
    def __init__(self, model, X_train, model_type='tree'):
        if model_type == 'tree':
            self.explainer = shap.TreeExplainer(model)
        elif model_type == 'linear':
            self.explainer = shap.LinearExplainer(model, X_train)
        else:
            # KernelExplainer for black-box models (slower)
            self.explainer = shap.KernelExplainer(
                model.predict_proba,
                shap.sample(X_train, 100)  # background dataset
            )
 
    def compute(self, X_sample: pd.DataFrame) -> dict:
        shap_values = self.explainer.shap_values(X_sample)
        return {
            'values': shap_values.tolist(),
            'base_value': float(self.explainer.expected_value),
            'feature_names': X_sample.columns.tolist(),
            'data': X_sample.values.tolist()
        }
 
    def by_group(self, X: pd.DataFrame, group_col: str) -> dict:
        results = {}
        for group, subset in X.groupby(group_col):
            results[str(group)] = self.compute(subset.drop(columns=[group_col]))
        return results

6.4 Counterfactual Generator
For individual decision explanations, PradnyaChakshu  generates counterfactual examples by flipping the value of each protected attribute and re-scoring the model. This surfaces statements like: "This applicant would have been approved if their gender were Male."

# engine/counterfactual.py
def generate_counterfactuals(model, row: pd.Series,
                              protected_attrs: list[str]) -> list[dict]:
    results = []
    original_pred = model.predict_proba([row])[0][1]
    for attr in protected_attrs:
        unique_vals = row[attr + '_candidates']  # pre-computed from dataset
        for alt_val in unique_vals:
            if alt_val == row[attr]: continue
            cf_row = row.copy()
            cf_row[attr] = alt_val
            cf_pred = model.predict_proba([cf_row])[0][1]
            results.append({
                'attribute':     attr,
                'original_value': row[attr],
                'counterfactual_value': alt_val,
                'original_score':  original_pred,
                'counterfactual_score': cf_pred,
                'score_delta': cf_pred - original_pred,
                'decision_changed': (
                    round(cf_pred) != round(original_pred)
                )
            })
    return sorted(results, key=lambda x: abs(x['score_delta']), reverse=True)

6.5 Risk Scoring Formula
The overall audit risk score (0–100) is a weighted composite of metric failures and disparity severities:

def compute_risk_score(metrics: list[MetricResult]) -> tuple[float, str]:
    weights = {
        'demographic_parity_difference': 0.25,
        'equalized_odds_difference':     0.20,
        'disparate_impact_ratio':        0.20,
        'equal_opportunity_difference':  0.15,
        'predictive_parity_difference':  0.10,
        'counterfactual_fairness_score': 0.10,
    }
    score = 0.0
    for m in metrics:
        w = weights.get(m.metric_name, 0.05)
        # Normalise severity: 0 = at threshold, 1 = 2x threshold
        severity = min(abs(m.overall_value) / m.threshold, 2.0) / 2.0
        score += w * severity * 100
    risk_level = (
        'LOW'      if score < 20  else
        'MEDIUM'   if score < 45  else
        'HIGH'     if score < 70  else
        'CRITICAL'
    )
    return round(score, 2), risk_level

7.4 Response Parsing & Fallback
import json, re
from anthropic import Anthropic
 
client = Anthropic()  # reads ANTHROPIC_API_KEY from environment
 
def call_llm_explainer(audit_result: dict) -> dict:
    prompt = build_llm_prompt(AuditResult(**audit_result))
    response = client.messages.create(
        model='claude-sonnet-4-20250514',
        max_tokens=2048,
        temperature=0.2,
        system=SYSTEM_PROMPT,
        messages=[{'role': 'user', 'content': prompt}]
    )
    raw = response.content[0].text
    # Strip any accidental markdown fences
    clean = re.sub(r'```json|```', '', raw).strip()
    try:
        return json.loads(clean)
    except json.JSONDecodeError:
        # Fallback: return structured template with error flag

            'causal_hypotheses': [],
            'remediation_steps': [],

8.3 WebSocket Hook
// hooks/useWebSocket.ts
export function useAuditWebSocket(auditId: string) {
  const [progress, setProgress] = useState<WSEvent | null>(null);
  useEffect(() => {
    const socket = io(WS_BASE_URL, {
      auth: { token: getAccessToken() },
      transports: ['websocket'],
    });
    socket.emit('subscribe', { audit_id: auditId });
    socket.on('audit_event', (event: WSEvent) => {
      setProgress(event);
      if (event.event === 'done') socket.disconnect();
    });
    return () => socket.disconnect();
  }, [auditId]);
  return progress;
}

SECURITY & PRIVACY REQUIREMENTS
Security & Privacy Requirements

9.1 Authentication & Authorisation
Control	Implementation
Auth Scheme	JWT Bearer tokens. Access token TTL: 15 minutes. Refresh token TTL: 7 days, HTTP-only cookie.
Password Storage	bcrypt hash, cost factor 12. Minimum 8 chars, 1 upper, 1 number required.
Role-Based Access	Roles: admin, analyst, viewer. Analysts can create audits; viewers are read-only; admins manage org settings.
Token Rotation	Refresh token is single-use; rotated on every use. Old refresh tokens invalidated on reuse (stolen token detection).
Rate Limiting	100 audit jobs/hour/org. 1000 API requests/hour/user. Enforced at Nginx layer and FastAPI middleware.
CORS	Strict origin allowlist. Credentials allowed only for own domain. No wildcard origins.

9.2 Data Security
Control	Implementation
Transport Encryption	TLS 1.3 enforced. HSTS header with max-age=31536000. Certificate pinning on mobile (future).
Data at Rest	PostgreSQL AES-256 transparent encryption (pgcrypto). S3 bucket encryption with KMS-managed keys.
Uploaded File Handling	Files stored with UUID filename (no original filename persisted). Virus scan via ClamAV on upload. Deleted after 24h via scheduled job.
Database Access	SQLAlchemy connection pool. Parameterised queries only (no string interpolation). Row-level security enforced via org_id foreign key.
Secrets Management	All credentials stored in environment variables. Never committed to VCS. Production uses Railway/Render secret store.
Audit Logging	All data access events logged to append-only audit_log table with user_id, action, resource_id, IP, timestamp.

9.3 Privacy Requirements
•	GDPR Article 25 (Privacy by Design): Uploaded datasets are never used for model training, never shared, and auto-deleted.
•	Data Minimisation: Only the columns necessary for the audit are processed; raw rows are never stored in PostgreSQL.
•	Right to Erasure: DELETE /audits/{id} removes all records, SHAP blobs, LLM outputs, and generated PDFs within 30 seconds.
•	Anonymisation: If auto-detection flags a column as a direct identifier (name, email, SSN pattern), PradnyaChakshu  masks it before processing.
•	Data Residency: Configurable per-org; EU orgs can be restricted to EU-hosted infrastructure.

9.4 LLM Security
•	No raw PII is included in Claude API prompts. Only aggregated statistics and column names are sent.
•	Prompt injection prevention: org_context field is HTML-escaped and truncated to 500 chars before inclusion in prompt.
•	Anthropic API key is stored in environment variable; never logged or returned to the client.
•	LLM responses are JSON-parsed and schema-validated before being stored; arbitrary text injection is rejected.

11.1 Scaling Strategy
•	Celery workers scaled horizontally to handle concurrent audit jobs. Each worker handles one job at a time.

ERROR HANDLING & LOGGING
Error Handling & Logging

13.1 Backend Error Handling Strategy
All unhandled exceptions in FastAPI are caught by a global exception handler that returns a structured error envelope and reports to Sentry with full context. Celery tasks use max_retries=3 with exponential backoff; after exhausting retries, the audit status is set to 'failed' and an error_message is stored.

# main.py — global exception handler
@app.exception_handler(Exception)
async def global_exception_handler(request: Request, exc: Exception):
    trace_id = sentry_sdk.capture_exception(exc)
    logger.error('Unhandled exception', extra={
        'trace_id': trace_id,
        'path': request.url.path,
        'method': request.method,
    })
    return JSONResponse(status_code=500, content={
        'error': {
            'code': 'INTERNAL_ERROR',
            'message': 'An unexpected error occurred.',
            'trace_id': str(trace_id),
        }
    })

13.3 Celery Task Error Handling
# tasks/fairness_tasks.py
@celery_app.task(
    bind=True,
    max_retries=3,
    default_retry_delay=2,  # seconds; doubles on each retry
    autoretry_for=(TransientError,),
    reject_on_worker_lost=True,
)
def compute_fairness_metrics(self, audit_id: str, file_path: str):
    try:
        # ... metric computation ...
        emit_ws_progress(audit_id, stage='metrics', pct=60)
    except PermanentError as e:
        # Non-retryable: mark audit failed immediately
        update_audit_status(audit_id, 'failed', str(e))
        raise
    except TransientError as e:
        # Retryable: Celery will retry automatically
        logger.warning(f'Transient error, retrying: {e}')
        raise self.retry(exc=e, countdown=2 ** self.request.retries)
